# verification-protocol.md — 산출물 결정론적 검증 프로토콜

## 개요
산출물의 품질을 AI 판단(확률론)에만 의존하지 않고,
기계적으로 확인 가능한 항목은 자동 검증(결정론)하는 프로토콜.

**핵심 원칙: "Keep Quality Left"**
- 기계가 확인할 수 있는 것은 기계에게 맡기고, AI는 판단만 하라
- 확률적 생성 → 결정론적 검증 → 에러가 AI의 다음 프롬프트 → 확률적 수정 → 결정론적 검증... 반복

---

## 1. 검증 분류: 결정론 vs 확률론

### 1-1. 결정론적 검증 (기계가 100% 확인 가능)
- 구조 검증: 필수 섹션 존재 여부
- 포맷 검증: 파일 형식, 마크다운 문법, 표 구조
- 수량 검증: 최소 줄 수, 단어 수, 항목 수
- 참조 검증: 내부 링크/참조 유효성
- 메타데이터 검증: frontmatter 필드 완전성

### 1-2. 확률적 검증 (AI 판단 필요)
- 내용 정확성, 톤 적합성, 창의성, 실용성
- → 기존 evaluation-protocol.md에 위임

---

## 2. 하네스별 검증 체크리스트

### 2-1. 공통 체크리스트 (모든 하네스)
□ 산출물 파일이 지정 경로에 생성되었는가
□ 파일이 비어있지 않은가 (최소 10줄)
□ 사용자 언어로 작성되었는가
□ 프로필 정보(이름, 회사)가 올바르게 반영되었는가
□ "초안(Draft)" 표기가 포함되었는가
□ 법률/재무/의료 하네스의 경우 면책 문구가 포함되었는가

### 2-2. 유형별 체크리스트

#### 전략/분석 문서 (.md)
□ 제목(H1) 존재
□ 목차 또는 섹션 구분 (H2 3개+)
□ 결론/요약 섹션 존재
□ 근거/출처 섹션 존재

#### 실행 문서 (.md / .docx)
□ 단계별 절차 포함
□ 체크리스트 또는 액션 아이템 포함
□ 담당자/일정 명시 (해당 시)

#### 데이터/수치 (.xlsx / .csv)
□ 헤더 행 존재
□ 데이터 행 1개 이상
□ 숫자 필드에 텍스트 혼입 없음

#### 프레젠테이션 (.pptx)
□ 제목 슬라이드 존재
□ 슬라이드 3장 이상
□ 빈 슬라이드 없음

## 3. 하이브리드 검증 루프

### 3-1. 루프 흐름

```
Step 1: 확률적 생성 (하네스 워크플로우 실행)
  ↓
Step 2: 결정론적 검증 (위 체크리스트 자동 수행)
  ↓
  IF 검증 통과 → Step 4
  IF 검증 실패 → Step 3
  ↓
Step 3: Lint-as-Instruction (실패 항목을 다음 프롬프트로 주입)
  "다음 항목이 누락되었습니다: [실패 목록]. 수정해 주세요."
  ↓ (수정 후 다시 Step 2)
Step 4: 확률적 평가 (evaluation-protocol.md 5차원)
  ↓
Step 5: 산출물 최종 전달
```

### 3-2. 반복 제한
- 최대 2회 반복 (무한 루프 방지)
- 2회 실패 시 현재 상태로 전달 + 실패 항목 사용자 보고

### 3-3. Lint-as-Instruction 포맷

검증 실패 시 AI에게 전달하는 메시지 형식:
```
[MoAI 검증 결과]
 ✅ 통과: 파일 생성됨, 최소 줄 수 충족, 언어 일치
 ❌ 실패: 결론 섹션 누락, 면책 문구 누락
 → 위 실패 항목을 보완하여 산출물을 수정하세요.
```

## 4. execution-protocol.md 연동

Phase 3(산출물 최종화) 이후, Phase 3.5(검증 루프)를 삽입:
```
Phase 3: 산출물 생성
Phase 3.5: 결정론적 검증 → (필요 시) 수정 루프
Phase 4: 확률적 평가 (기존 evaluation-protocol)
Phase 5: 반성 & 진화
```
